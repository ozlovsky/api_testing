### ПРАКТИКА 12: Ошибки в XML: 
Иногда в тестовых заданиях встречаются задачки на проверку ошибок синтаксиса XML.
- Проверить следующий XML и прислать список найденных ошибок в качестве ответа

```
  <?xml version="1.0" encoding="UTF-8"?>
  <note date="12/05/2022">
  <to><name>Alena</to></name>
  <lastname>Petrova</lastname>
  <from>Alex</From>
  <update=12/05/2022><update>
  </note>
```
 Мой Ответ: 
```
<?xml version="1.0" encoding="UTF-8"?> 
<note date="12/05/2022">                 <!-- Формат Дата\Текст. Решил оставить текст. -->
<to><name>Alena</name></to>              <!-- Очерёдность закрытия тега (вложение) -->
<to><lastname>Petrova</lastname></to>    <!-- Желательно поместить в <to>, иначе теряется контекст -->
<from>Alex</from>                        <!-- Регистр должен быть одинаковым для открывающего и закрывающего тега -->
<update date="12/05/2022"></update>      <!-- Опять вопрос к формату + отсутствие закрытого тега -->
</note>
```

### ПРАКТИКА 13: Создание REST-коллекции:
Давайте создадим свою первую коллекцию для "Интернет-магазин" и начнем погружение в тестирование backend.
- Скачайте Postman.
- Создать коллекцию "DemoShopping".
- Создать папку "Products" и "Cart"и оформить все методы, которые относятся к этим категориям в Swagger.
- В методе PUT для обновления товара уберите из тела запроса id, иначе будет ошибка.
- Не забывайте, что для запросов к корзине вам понадобится настроить авторизацию. Это отдельный метод в Users.
- При создании проверок основывайтесь в первую очередь на статус-коды.
- Не забывайте, что названия запросов должны быть понятными и отражать суть проверки.
- Создайте переменные окружения QA, которые будут часто переиспользоваться, например, базовый URL, token и ID. Подумайте, что еще можно добавить в них.
- Если вы будете использовать переменную token, то назовите ее key. Это нужно для того, чтобы у коллекции в будущем не слетела публичность.
- Для всех методов Products напишите автотесты, которые могут проверять: статус-код после отправки запроса (обязательный тест), проверки для тела (тип данных в значениях, изменение ключа и значения для POST и т.д.), время ответа и т.д.
- Настоятельно рекомендую не просто импортировать готовый JSON с коллекциями или CURL, а сделать это задание в ручном режиме. Так вы больше разберётесь в инструментах и функциях Postman.
- Перед началом и после окончания работы со Swagger запустите метод "Сброс базы данных". Он удалит лишние созданные продукты, заказы и очистит корзину. Также этот скрипт автоматически запускается в 5 утра GMT+3.
- В качестве решения пришлите ссылку на свою публичную (!) коллекцию, иначе я не смогу ее просмотреть.

### ПРАКТИКА 14: Создание SOAP-коллекции
Иногда на проектах используют протокол SOAP.

Тестировать запросы для таких сервисов можно в инструментах Postman или SoapUI.

Предлагаю выбрать первый вариант и создать коллекцию в Postman для тестирования данного <a href="http://webservices.oorsprong.org/websamples.countryinfo/CountryInfoService.wso?WSDL">сервиса</a>, который позволяет получать информацию о целевой стране.

<a href="https://en.wikipedia.org/wiki/List_of_ISO_3166_country_codes">Список ISO-кодов</a> стран для тестирования методов.

В коллекцию должны попасть следующие методы:
- ListOfContinentsByName
- CountryName
- FullCountryInfo
- LanguageName
